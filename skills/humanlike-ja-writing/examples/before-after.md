# Before / After Examples

## 1. 一般論だけの効率化

Bad:

```text
AIを活用することで、業務効率化が期待できます。
```

Good:

```text
議事録の整形だけなら、AIに任せたほうが早かった。人が30分かけていた作業が、レビュー込みで10分ほどになった。
```

Why:

- 「業務効率化」を、議事録整形という具体作業に落とした。
- Before/Afterの時間を入れた。
- 断言範囲を「議事録の整形だけ」に限定した。

## 2. 無生物主語と抽象語

Bad:

```text
この結果は、ユーザー体験の重要性を浮き彫りにしています。
```

Good:

```text
5人中4人が検索フォームの場所で止まった。初回利用者には、今の配置は見つけにくい。
```

Why:

- 「結果」を主語にせず、実際に起きたことを書いた。
- 「重要性」「浮き彫り」を削り、判断を具体化した。

## 3. テンプレ導入

Bad:

```text
本記事では、AIっぽくない文章を書くためのポイントを解説します。
```

Good:

```text
AIに「自然に書いて」と頼むだけでは、だいたい自然にならない。
```

Why:

- 構成宣言ではなく、主張から入った。
- 読者が知りたい問題を一文で出した。

## 4. 因果が薄い

Bad:

```text
これにより、チーム全体の生産性向上につながります。
```

Good:

```text
レビュー前の整形作業を減らせるので、チームは中身の確認に時間を使える。
```

Why:

- 「これにより」を削った。
- 何が減り、何に時間を使えるのかを書いた。

## 5. 捏造された人間味

Bad:

```text
実際に使ってみると、思わず嬉しくなる体験でした。
```

Good:

```text
実際の利用体験がないなら、体験談としては書かない。機能の説明に留める。
```

Why:

- 人間味を出すための架空体験を禁止した。
- 自然さより正確さを優先した。

## 6. 定型結論

Bad:

```text
いかがでしたか。この記事が参考になれば幸いです。
```

Good:

```text
AI臭を消すなら、最後に語尾を直すより、最初に主張と具体材料を渡したほうがいい。
```

Why:

- 儀礼的な締めを削った。
- 書き手の判断で終えた。
